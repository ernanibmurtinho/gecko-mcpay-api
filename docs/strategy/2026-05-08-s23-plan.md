# S23 Plan — Live Marketplace Spend + Source Routing Reachability

**Window:** 2026-05-08 → 2026-05-13 (6 days)
**Predecessor:** S22 (neobank corpus + hosted MCP at v0.2.24)
**Author:** staff-engineer
**Theme:** Make the seeded corpus actually load at query time, and prove live x402 spend against pay.sh + Bazaar from production. Close the wedge-reachability gap (FIX-12) before any public "we use pay.sh/Bazaar" claim.

---

## 1. Context

S22 shipped the spine: 298 chunks live in Mongo across `paysh_manifest`, `bazaar_manifest`, and web seeds (N5); the hosted MCP at `api.geckovision.tech/mcp/` exposes 19 tools (v0.2.24); `paysh_live.py` and `bazaar_live.py` caller modules are merged but their tests are gated under `live_paysh` / `live_bazaar` markers and have not yet executed real outbound x402 spend from production. The neobank example app and skill.md updates are queued in `gecko-claude` PRs awaiting merge.

The honest read of where we are: S22 built the *ingestion* loop. It did not build the *runtime retrieval* loop. Caio's last three sessions (`bbae8c8a`, `7e8499e5`, `e514e164`) confirm that `gecko_research` still routes mostly to Tavily web sources, even on neobank queries where 200+ paysh/bazaar chunks exist. This is the textbook trap from `feedback_wedge_reachability_check`: dispatcher fires, ledger increments, per-layer tests pass, but the bytes never reach the LLM context. **FIX-12 is the top of the sprint.**

S23 is the unlock for the public competitor framing in `project_real_competitors_2026_05_07`: Gecko competes with frames.ag / Bazaar / Apify by offering categorized retrieval *plus* verdict synthesis on top of those marketplaces. That story is only honest once (a) seeded corpus actually routes at query time, and (b) Gecko is provably executing live paid calls against pay.sh and Bazaar from production. Without both, the demo is a curated set of Tavily citations dressed in marketplace language.

- **G1 — Reachability green:** A prod `gecko_research` call against a neobank prompt returns a session whose `sources[]` includes ≥1 chunk tagged `paysh_manifest` and ≥1 tagged `bazaar_manifest`. Verified via curl in §8.
- **G2 — Live spend recorded:** ≥3 live x402 settlements against pay.sh providers and ≥3 against Bazaar services, recorded as VCR-style fixtures and as `paysh_live` / `bazaar_live` chunks in Mongo. Total spend ≤ $1.00 USDC across the sprint.
- **G3 — Eval gate locked:** `CORPUS-DENSITY-EVAL` runs in CI; for any vertical with ≥100 chunks seeded, the eval fails if `gecko_research` returns 0 non-Tavily citations. Replicates the holdout-live shape from `feedback_eval_harness_rag_gap`.

## 5. Falsifiers

1. **T1 reveals the dispatch graph needs structural rework, not a wire fix.** If by Day 2 EOD the audit finds the retrieval router can't filter on `provider_kind` without a Mongo schema migration, FIX-12 grows beyond M-L. **Pivot:** ship a narrow router patch that hard-prepends top-K paysh/bazaar chunks for any neobank/dex query (v1_block style), defer the proper filter to S24. Honesty win, technical-debt take.
2. **Live x402 spend against pay.sh or Bazaar fails in prod.** If T2/T3 hit `WWW-Authenticate` propagation issues, signature rejection, or the buyer-wallet env path is not yet provisioned by operator (§9), live spend can't run from ECS. **Pivot:** record fixtures from operator's local dev wallet, ship as "live-recorded" rather than "live-from-prod"; defer prod buyer-wallet to S24. Public framing must reflect this honestly.
3. **Reachability eval (T4) green but verdict quality regresses.** If injecting paysh/bazaar chunks into the LLM context degrades verdict quality (noisy manifest entries crowd out signal), G1 passes but the product gets worse. **Pivot:** add a per-source weight in the retrieval ranker, drop manifest chunks below a relevance threshold. Test holdout-live deltas before merging the weighting.

See `~/.claude/projects/-home-nan-PycharmProjects-Gecko-gecko-mcpay-api/memory/project_buyer_wallet_blocker_2026_05_08.md` for the full 4-step checklist:

1. **Provision a dedicated buyer-side Solana wallet.** Currently the prod wallet only signs INBOUND payments to Gecko's services (the receive-side x402 flow). To call OUTBOUND paid services on pay.sh and Bazaar, Gecko itself needs a buyer wallet with USDC balance. This MUST be a separate keypair from the receive-side wallet — security boundary.
2. **Fund the buyer wallet with ~$5 USDC** (Solana mainnet). $1 covers the sprint; remainder is buffer for ongoing nightly probes through S24.
3. **Populate `BUYER_WALLET_KEYPAIR` SSM secure-string parameter** at `/gecko-api/BUYER_WALLET_KEYPAIR` (region us-east-2). Update ECS task definition to mount this env var. Confirm via `aws ssm get-parameter --name /gecko-api/BUYER_WALLET_KEYPAIR --with-decryption` (do not paste value into chat — verify locally).
4. **Confirm the existing `EMBED_MODEL` SSM param is `voyage-3-large`** per the S22 wake-up note; the seeded chunks were embedded with that model.
5. **Approve the buyer-wallet spend cap policy:** $0.50 per provider-set per nightly run, hard-capped at the application layer in T2/T3. Founder ack required before nightly cron is enabled.

Until items 1-3 are done, T2 and T3 cannot run from production — they can still complete their *recorded-fixture* deliverable from operator's local dev wallet, satisfying G2 on the artifact axis but not the "from prod" axis. This is acceptable for sprint sign-off if explicitly noted.
